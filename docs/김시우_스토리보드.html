1. SC-F-01 | 앱 시작 및 사용자 식별

앱 최초 실행 시 device_id를 localStorage에서 로드하거나 UUID v4로 신규 생성하고, 서버에 등록하여 user_id를 획득한다. 이후 tbl_user_preferences에서 선호도를 불러와 filterStore 초기값으로 설정한다.
앱 진입
→
device_id 로드/생성
→
POST /api/users
→
userStore 저장
→
GET /api/users/{device_id}/preferences
→
filterStore 초기화
Step액터/모듈동작비고
1앱 초기화 모듈localStorage에서 device_id 조회. 없으면 UUID v4 신규 생성 후 저장최초 실행 여부 구분
2AxiosPOST /api/users { device_id } 요청tbl_users UPSERT
3Zustand StoreuserStore.setUserId(response.user_id) 호출이후 모든 API 요청에 user_id 사용
4AxiosGET /api/users/{device_id}/preferences 요청tbl_user_preferences 조회
5Zustand StorefilterStore를 서버 선호도 값으로 초기화 (category, radius, sort)신규 사용자: 서버 기본값(전체, 1000m, distance) 적용
6앱홈 화면 진입선호도 기반 필터 초기값으로 식당 목록 즉시 조회

2. SC-F-02 | 위치 수집 및 식당 조회

useGeolocation Hook으로 GPS 좌표를 수집하여 locationStore에 저장하고, /api/geocode로 행정동을 역지오코딩한 뒤 filterStore 구독 하에 /api/restaurants로 식당 목록을 조회한다.
useGeolocation Hook
→
locationStore
→
GET /api/geocode
→
filterStore 구독
→
GET /api/restaurants
→
UI 리렌더링

2-1. 정상 흐름 - 위치 수집 및 역지오코딩

Step액터/모듈동작비고
1useGeolocation Hook훅 마운트 시 권한 요청 트리거navigator.geolocation.getCurrentPosition()
2UI권한 요청 중 로딩 상태 표시Zustand: status = "loading"
3useGeolocation HookGPS 측위 성공 - { lat, lng } 수신GeolocationPosition 객체 파싱
4Zustand StorelocationStore.setLocation({ lat, lng }) 호출전역 상태 업데이트
5AxiosGET /api/geocode?lat=&lng= 요청프록시 서버 경유, Redis 캐시 우선 조회
6Axios응답 수신 { region, fullAddress }200 OK
7Zustand StorelocationStore.setRegion("역삼동") 호출헤더 UI에 행정동 표시 반영

2-2. 정상 흐름 - 식당 목록 조회

Step액터/모듈동작비고
1Zustand StorefilterStore: { category, radius, sort } 상태 변경 감지필터 칩 탭 시 트리거
2useRestaurants Hook필터 상태 구독 - API 재요청 트리거Zustand selector로 필터값 구독
3Zustand StorerestaurantStore.setLoading(true)스켈레톤 UI 표시
4AxiosGET /api/restaurants?lat=&lng=&category=&radius=&sort= 요청Axios 인스턴스 baseURL 설정
5Axios응답 수신 { restaurants: [...], total, cachedAt }200 OK
6Zustand StorerestaurantStore.setRestaurants(data.restaurants)전역 상태 업데이트
7UI카드 리스트 리렌더링. 각 카드의 즐겨찾기 아이콘은 favoritesStore를 참조하여 채움/빈 상태 표시React 리렌더링 트리거

2-3. 예외 흐름 - GPS 권한 거부 또는 타임아웃

GPS 권한 거부 및 타임아웃은 별도 에러 유형으로 분기 처리하며, 각각 적합한 fallback UI를 제공한다.
Step액터/모듈동작비고
1useGeolocation HookPermissionDeniedError 또는 timeout 발생timeout 기준: 10초
2Zustand StorelocationStore.setError("permission_denied")에러 상태 전역 저장
3UI수동 지역 입력 fallback UI 렌더링GPS 거부 안내 + 수동 입력 필드
4Axios수동 입력 지역명으로 검색 API 요청GET /api/restaurants?region=역삼동

2-4. 예외 흐름 - 네트워크 오류 및 재시도

Step액터/모듈동작비고
1Axios네트워크 오류 또는 5xx 에러 발생interceptor에서 catch
2Axios Interceptor자동 재시도 1회 (retry logic)503일 경우 재시도 없이 바로 에러
3Zustand StorerestaurantStore.setError(errorMessage)에러 유형별 메시지 분기
4UI스켈레톤 → ErrorState 컴포넌트 전환"재시도" 버튼 제공

3. SC-F-03 | 식당 선택 및 이력 저장

식당 카드 탭 시 플랫폼에 따라 딥링크를 호출하고, 동시에 POST /api/history를 fire-and-forget으로 전송하여 tbl_selection_history에 이력을 저장한다.
식당 카드 탭
→
플랫폼 감지
→
Android: intent://
/
iOS: nmap://
/
fallback URL
+
POST /api/history
→
tbl_selection_history INSERT

3-1. 딥링크 처리 흐름

Step액터/모듈동작비고
1UI식당 카드 탭 - naverLink 값 추출restaurantStore에서 선택된 식당 정보
2딥링크 모듈플랫폼 감지 (Android / iOS / 기타)navigator.userAgent 파싱
3a딥링크 모듈[Android] intent:// 스킴으로 네이버 앱 호출intent://map.naver.com/...
3b딥링크 모듈[iOS] nmap:// universal link 호출nmap://place?id=xxxxx
4딥링크 모듈앱 미설치 또는 딥링크 실패 시 fallback URL로 전환window.location.href = naverLink (모바일 웹)

3-2. 선택 이력 저장 흐름

Step액터/모듈동작비고
1UI식당 카드 탭 - 선택된 식당 정보 추출 (place_id, place_name, category)restaurantStore에서 데이터 참조
2AxiosPOST /api/history { user_id, place_id, place_name, category, region } 요청 (비동기, 응답 대기 없이 fire-and-forget)딥링크 호출과 병렬 처리
3서버tbl_selection_history INSERT 후 카테고리별 빈도 집계 → tbl_user_preferences.preferred_category 자동 갱신서버 비동기 처리

4. SC-F-04 | 즐겨찾기 관리

홈 화면 카드의 즐겨찾기 아이콘 탭 시 현재 상태에 따라 추가 또는 삭제를 처리한다. Optimistic Update로 UI를 먼저 변경 후 API 요청하며, 실패 시 rollback한다.
즐겨찾기 아이콘 탭
→
favoritesStore 상태 확인
→
Optimistic Update
→
POST 또는 DELETE
→
성공/실패 처리

4-1. 즐겨찾기 추가 흐름

Step액터/모듈동작비고
1UI즐겨찾기 아이콘 탭 - favoritesStore에서 해당 place_id 존재 여부 확인미등록 상태 확인
2Zustand StorefavoritesStore.addOptimistic(restaurant) - 즉시 아이콘 채움 상태로 변경Optimistic Update
3AxiosPOST /api/favorites { user_id, place_id, place_name, category, naver_link } 요청tbl_favorites INSERT
4aAxios201 Created - favoritesStore.confirmAdd(favorite_id) 호출서버 favorite_id를 store에 저장
4bAxios실패 시 favoritesStore.rollback(place_id) 호출 - 아이콘 빈 상태로 복구409 중복은 무시 처리

4-2. 즐겨찾기 삭제 흐름

Step액터/모듈동작비고
1UI즐겨찾기 아이콘 탭 - favoritesStore에서 해당 place_id의 favorite_id 조회등록 상태 확인
2Zustand StorefavoritesStore.removeOptimistic(place_id) - 즉시 아이콘 빈 상태로 변경Optimistic Update
3AxiosDELETE /api/favorites/{favorite_id} 요청tbl_favorites DELETE
4aAxios200 OK - favoritesStore.confirmRemove(place_id) 호출store에서 항목 완전 제거
4bAxios실패 시 favoritesStore.rollback(place_id) 호출 - 아이콘 채움 상태로 복구

5. SC-F-05 | 에러 처리

Axios interceptor 중앙 집중 에러 처리, GPS 거부/타임아웃 fallback, 503/네트워크 오류 처리, WebView Bridge 연동을 통합하여 관리한다.
API 요청
→
Axios Interceptor
→
에러 유형 분기
→
GPS 오류 / 503 / 네트워크 오류
→
fallback UI / WebView Bridge

5-1. Axios interceptor 에러 처리 흐름

Step액터/모듈동작비고
1Axios Interceptor모든 API 응답을 response interceptor에서 일괄 수신axios.interceptors.response.use()
2Axios Interceptor4xx 에러 - 에러 코드별 메시지 분기 처리401: 재인증, 404: 리소스 없음
3Axios Interceptor5xx 에러 중 503 Service Unavailable - 재시도 없이 즉시 에러 반환점검 중 안내 메시지 표시
4Axios Interceptor503 외 5xx 및 네트워크 오류 - 자동 재시도 1회 후 실패 시 에러 반환exponential backoff 미적용 (MVP)
5Zustand Store에러 유형에 따라 해당 store의 setError() 호출restaurantStore / locationStore 등
6UIErrorState 컴포넌트 렌더링 - "재시도" 버튼 제공스켈레톤 → 에러 UI 전환

5-2. GPS 거부/타임아웃 에러 흐름

GPS 권한 거부 및 타임아웃은 Axios interceptor가 아닌 useGeolocation Hook 내부에서 별도 처리한다.
Step액터/모듈동작비고
1useGeolocation HookPermissionDeniedError (code: 1) 또는 timeout (code: 3) 발생timeout 기준: 10초
2Zustand StorelocationStore.setError("permission_denied" | "timeout") 호출에러 코드로 유형 구분
3UI수동 지역 입력 fallback UI 렌더링GPS 거부 안내 + 수동 입력 필드
4Axios수동 입력 지역명으로 식당 검색 요청GET /api/restaurants?region=역삼동

5-3. WebView JavaScript Bridge 연동

WebView Bridge 규격은 Flutter/React Native 팀원과 인터페이스를 사전 합의한다. MVP에서는 Android intent://, iOS universal link 방식을 우선 구현하고, Bridge는 점진적으로 적용한다.
Step액터/모듈동작비고
1WebView (Flutter/RN)앱 시작 시 JavaScript Bridge 주입window.nativebridge 객체 등록
2Web → Native딥링크 실행 시 nativebridge.openNaver(url) 호출네이티브 앱 레벨에서 인텐트 처리
3Native → WebGPS 좌표 변경 시 bridge.postMessage({ lat, lng }) 전달WebView 내 JS eventListener 수신
4Zustand Store수신한 좌표로 locationStore 업데이트화면 자동 갱신
