이슈 관리 및 해결 과정

* 발생한 이슈 (Issue):

  - FE 로컬 서버(localhost:5173)에서 BE 로컬 서버(localhost:3000)로 API 호출 시 브라우저에서 CORS Preflight (OPTIONS) 오류가 발생하며 요청이 차단됨.


* 원인 분석:

  - NestJS 기본 설정에서 특정 Origin을 명시하지 않았거나, 브라우저가 사전 요청(Preflight)으로 보내는 `OPTIONS` 메서드에 대한 허용 설정이 누락됨.

  - `Content-Type` 등 커스텀 헤더가 포함된 요청에 대해 `Access-Control-Allow-Headers` 설정이 부족함.


* 해결 과정 및 결과 (Resolution):

  - NestJS의 `app.enableCors()` 옵션에 객체를 주입하여 세부 설정 명시화.

  - `methods: ['GET', 'POST', 'PATCH', 'DELETE', 'OPTIONS']` 로 OPTIONS 메서드를 명시적으로 허용.

  - `allowedHeaders`에 `Content-Type`을 추가하고, `origin` 속성을 환경변수로 관리하여 개발/운영 환경 분리.

  - 결과: MDN Web Docs의 CORS 정책 문서를 참고하여 근본적인 설정을 맞췄으며, FE 팀의 로컬 API 연동 테스트가 정상적으로 통과함 (응답 200 OK 확인).

* Keep (유지할 점):

  - Docker를 통한 환경 통일. FE 팀원들이 복잡한 백엔드 설정 없이 컨테이너만 띄워서 바로 연동 테스트를 할 수 있었던 점이 개발 속도 향상에 크게 기여함.

  - 외부 API 호출 로직을 Wrapper로 추상화하여, 실제 비즈니스 로직(Service 계층)에서 분리한 설계 구조.


* Problem (문제점/아쉬운 점):

  - Rate Limit 미들웨어 설정을 적용했으나 현재 in-memory 방식을 사용 중임. EC2 배포 후 서버 프로세스가 재시작되면 카운터가 초기화되는 문제가 예상됨.

  - 로깅 시스템이 `console.log`에만 의존하고 있어, 프로덕션 환경에서의 에러 추적이 어려움.


* Try (시도할 점/개선 방향):

  - Rate Limit 카운터를 in-memory에서 Redis 기반으로 교체(`rate-limit-redis` 패키지 활용 등)하여 분산 환경 및 재시작에도 카운트가 유지되도록 구조 개선.

  - `winston` 또는 `nestjs-pino` 같은 로거를 도입하여 파일 저장 및 로그 레벨 세분화 작업 진행.
