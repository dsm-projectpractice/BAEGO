* 주요 개발 내용

  1. 실제 서버 API 연동

    - Mock JSON을 제거하고 백엔드(이동현)가 배포한 프록시 서버 엔드포인트로 전환.

    - `useRestaurants` 훅 내부의 fetch 로직을 Axios 인스턴스 기반으로 교체하고, 요청 파라미터(lat, lng, category, radius)를 Zustand store에서 주입하는 구조로 리팩토링.

    - API 응답의 카멜케이스 필드명 확정에 따라 타입 정의 일부 수정(distanceM → distance 등).


  2. 네이버 지도 딥링크 구현

    - 식당 카드 클릭 시 네이버 지도 앱(nmap://) 또는 모바일 웹(map.naver.com)으로 연결되는 딥링크 브릿지 함수 `openNaverMap` 구현.

    - 앱이 설치되지 않은 환경을 고려하여 500ms 타임아웃 후 웹 fallback으로 전환하는 로직 적용.


  3. Axios Interceptor 및 통합 에러 처리

    - Request Interceptor: 요청마다 공통 헤더(Content-Type) 자동 주입.

    - Response Interceptor: HTTP 4xx/5xx 응답을 가로채 통합 에러 객체로 변환 후 Toast 알림 노출.

    - 네트워크 단절 상태(Network Error) 감지 시 "인터넷 연결을 확인해주세요" 토스트 메시지 표시.


※ 첨부 스크린샷:

// src/lib/axiosInstance.ts import axios from 'axios'; import { toast } from '../components/Toast'; const axiosInstance = axios.create({ baseURL: import.meta.env.VITE_API_BASE_URL, timeout: 8000, }); axiosInstance.interceptors.request.use((config) => { config.headers['Content-Type'] = 'application/json'; return config; }); axiosInstance.interceptors.response.use( (response) => response, (error) => { if (!error.response) { toast.error('인터넷 연결을 확인해주세요.'); } else if (error.response.status >= 500) { toast.error('서버 오류가 발생했습니다. 잠시 후 다시 시도해주세요.'); } else if (error.response.status === 429) { toast.error('요청이 너무 많습니다. 잠시 후 다시 시도해주세요.'); } return Promise.reject(error); } ); export default axiosInstance;

이슈 관리 및 해결 과정

* 발생한 이슈 (Issue):

  - 실제 API로 교체 후 식당 목록이 간헐적으로 빈 배열로 반환되는 현상 발생. 특히 반경 500m 필터 적용 시 결과가 0건으로 나오는 케이스가 빈번함.


* 원인 분석:

  - Zustand store의 초기 radius 값(1000)이 컴포넌트 마운트 직후 API 호출 시점에 정상 반영되지 않는 경우가 있었음. React의 렌더링 사이클과 Zustand 상태 초기화 타이밍이 맞지 않아 radius 파라미터가 undefined로 요청되는 케이스 확인.

  - 추가로 백엔드 Naver API 반경 파라미터 단위가 미터(m)가 아닌 킬로미터(km)였던 것을 클라이언트에서 m 단위로 전달하고 있었음. (API 명세 오해)


* 해결 과정 및 결과 (Resolution):

  - Zustand store 초기값에 명시적 기본값(radius: 1000, category: 'all')을 선언하고, API 호출 훅 내부에서 파라미터 유효성 검사 후 기본값 fallback 처리 추가.

  - 백엔드 담당자(이동현)와 API 명세 재확인 후 radius 단위를 m → km로 변환하여 전달하도록 `useRestaurants` 훅 내 변환 로직(radius / 1000) 추가.

  - 결과: 500m 필터 적용 시에도 주변 식당이 정상적으로 조회되며, 빈 배열 발생 케이스 해소.
KPT 회고 및 차주 계획

  - API 연동 전환 시 타입 정의부터 수정하고 컴파일 오류를 따라가며 수정하는 방식이 빠른 전환에 효과적이었음.


* Problem (문제점/아쉬운 점):

  - 즐겨찾기 데이터를 localStorage에만 저장하는 현재 구조는 기기가 바뀌거나 브라우저 데이터 삭제 시 초기화됨. 서버 연동 필요성을 느꼈으나 MVP 범위 상 백엔드 즐겨찾기 API 연동은 우선순위를 낮춰 다음 주기로 이월.


* Try (시도할 점/개선 방향):

  - React Query(TanStack Query) 도입을 검토하여 API 캐싱 및 리패치 로직을 선언적으로 관리하는 방향으로 리팩토링.
