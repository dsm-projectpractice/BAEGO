## 2. 기획 배경 및 목적

### 해결하고자 하는 문제
점심·저녁 식사 시 메뉴 선택에 소요되는 의사결정 비용(결정 장애)은 일상적으로 반복되지만, 기존 서비스는 검색어 입력을 전제로 하여 능동적 탐색을 요구한다. **위치 자동 감지 기반의 수동적(passive) 추천 흐름**이 필요하다.

### 기획 의도 및 목표
네이버 지역 검색 API를 프록시 서버로 경유하여 신뢰도 높은 식당 데이터를 수집하고, 터치 한 번으로 메뉴를 결정할 수 있는 **최소 기능 제품(MVP)**을 구축한다. 이후 사용자 취향 데이터 축적을 기반으로 개인화 추천 엔진으로 고도화하는 것을 장기 목표로 설정한다.

## 3. 서비스 상세

### 타겟 사용자
- 점심·저녁 메뉴 결정에 어려움을 겪는 직장인 및 대학(원)생
- 낯선 지역에서 식당 탐색이 필요한 여행자

### 핵심 기능 (Core Features)

| 기능 | 상세 설명 |
|------|-----------|
| 위치 기반 식당 탐색 | HTML5 Geolocation API로 사용자 위경도 수집 후 Naver Reverse Geocoding으로 행정동 단위 주소 변환, 상단 UI 표시 |
| 메뉴 추천 엔진 | 현재 운영 중인 식당 중 카테고리별 무작위 큐레이션. 거리순(500m/1km/2km), 평점순, 카테고리(한/중/일/양식) 필터 제공 |
| 상세 정보 딥링크 | 식당 선택 시 네이버 플레이스 앱 또는 모바일 웹으로 직접 연결, 현재 위치 기준 길 찾기 연동 |

### 차별점

| 구분 | 기존 서비스 (네이버 지도, 카카오맵 등) | 본 서비스 |
|------|--------------------------------------|-----------|
| 탐색 방식 | 검색어 입력 또는 카테고리 탭 선택 필요 | 입력 없이 위치 감지만으로 즉시 추천 |
| 정보 제공 | 방대한 정보로 인한 선택 피로도 증가 | 단일 큐레이션 결과 제시로 결정 비용 최소화 |
| UX 패턴 | 능동적 탐색 전제 UX | 수동적(passive) 추천 UX로 사용자 이탈 방지 |

## 8. 기대 효과 및 발전 방향

### 기대 효과
- **개발 효율성**: 네이버의 검증된 DB를 활용하여 식당 정보 수집·갱신 비용 절감, 데이터 신뢰성 확보
- **보안 안정성**: 프록시 서버 구조로 API 키 클라이언트 노출 차단, Redis 인메모리 캐싱으로 일일 API 쿼터 절약
- **사용자 편의성**: 검색어 입력 없이 터치 한 번으로 식사 메뉴 결정 프로세스를 단축하여 결정 피로도 해소

### 추후 발전 방향
- 사용자의 선택 이력 및 평가 데이터를 축적하여 개인화 AI 추천 모델로 고도화
- 동행인 수 및 예산 입력 기반의 그룹 추천 기능 추가
- 앱스토어(Google Play / App Store) 정식 배포 및 푸시 알림 연동

## 9. 기술적 고려사항 및 해결 전략

| 구분 | 리스크 | 해결 전략 |
|------|--------|-----------|
| API 보안 | Client ID / Secret 클라이언트 노출로 인한 키 탈취 위험 | 서버사이드 Environment Variable (.env)로 키 관리, 클라이언트에서 직접 Naver API 호출 구조적 차단 |
| API 쿼터 | 동일 지역 반복 요청으로 일일 호출 한도 조기 소진 | Redis 캐싱 적용, 지역·카테고리 조합 키 기반 TTL 설정으로 데이터 신선도와 쿼터 효율 균형 유지 |
| 서버 가용성 | EC2 단일 인스턴스 장애 시 서비스 중단 | PM2 프로세스 매니저로 서버 자동 재시작, GitHub Actions 배포 실패 시 이전 버전 자동 롤백 설정 |
| CI/CD 안정성 | 잦은 푸시로 인한 불안정한 배포 | GitHub Actions에 lint·build 단계 필수화, main 브랜치 PR merge 시에만 자동 배포 트리거 설정 |
