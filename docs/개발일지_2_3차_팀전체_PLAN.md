## 2. 2차 개발일지 작성 계획 (3~4주차)

### 팀 전체 목표
- 백엔드 기본 서버 구축 및 네이버 API 연동 모듈화
- 프론트엔드 핵심 UI 페이지 컴포넌트화 및 위치 정보 수집(Geolocation) 기능 구현
- FE-BE 간 Mock 데이터를 활용한 1차 연동 테스트

### 팀원별 작성 내용 (Role & Responsibilities)

#### 👦 이동현 (PM & Backend)
- **주요 내용:**
  - Docker / NestJS(Express) 개발 환경 초기 세팅
  - 네이버 Search, Reverse Geocoding API Wrapper 모듈 개발 완료
  - `.env` 기반 환경변수(API Key) 설정 및 CORS 처리
- **이슈 포인트 (예상):** 프론트엔드 로컬 연동 시 CORS Preflight 에러 해결 과정 작성

#### 👦 조성현 (FE UI/UX)
- **주요 내용:**
  - Figma 기반 추천 결과 화면, 필터 팝업 등 전체 UI 마크업 완료
  - Tailwind CSS를 활용한 공통 컴포넌트(Button, Card, BottomSheet) 구현
  - 모바일 웹뷰 환경에 맞춘 반응형 레이아웃 적용
- **이슈 포인트 (예상):** 디바이스별 상이한 뷰포트 높이(vh) 이슈 및 모바일 브라우저 하단 바 가림 현상 해결 과정

#### 👦 김시우 (FE 기능/연동)
- **주요 내용:**
  - `useGeolocation` 커스텀 훅 개발 (위도/경도 추출)
  - 카테고리, 거리 등 클라이언트 전역 상태 관리 세팅 (Context 또는 Zustand)
  - BE에서 제공한 Mock 데이터를 활용한 UI 바인딩 테스트
- **이슈 포인트 (예상):** 위치 정보 권한 거부 시의 예외 처리(Fallback) 로직 및 사용자 안내 UX 구현 과정

## 3. 3차 개발일지 작성 계획 (5~8주차 / 최종)

### 팀 전체 목표
- Redis 캐싱을 통한 네이버 API 쿼터 방어 및 성능 최적화
- FE-BE 실 API 연동 완료 및 딥링크/웹뷰 브릿지 연결
- 최종 QA, 예외 처리 고도화 및 AWS EC2 프로덕션 배포 완료

### 팀원별 작성 내용 (Role & Responsibilities)

#### 👦 이동현 (PM & Backend / Infra)
- **주요 내용:**
  - Redis 캐싱 레이어 도입 및 Rate Limit 구현 (API 쿼터 방어)
  - 예외 처리(Global Exception Filter) 및 표준 에러 코드 적용
  - AWS EC2 단일 인스턴스 배포 및 GitHub Actions CI/CD 파이프라인 구축
- **이슈 포인트 (예상):** 배포 자동화 중 발생한 포트 충돌 문제 및 Redis 캐시 히트율(Cache Hit Ratio) 개선 과정

#### 👦 조성현 (FE UI/UX)
- **주요 내용:**
  - 데이터 로딩 스켈레톤(Skeleton) UI 및 스피너 추가
  - 페이지 간 화면 전환 애니메이션(Transition) 적용
  - 최종 디자인 QA 수행 및 미세 UI 폴리싱 (버튼 햅틱 피드백 등)
- **이슈 포인트 (예상):** 애니메이션 적용 후 구형 모바일 기기에서의 렌더링 성능 저하 프레임 개선

#### 👦 김시우 (FE 기능/연동)
- **주요 내용:**
  - BE 실 API 연동 (식당 리스트 GET, 사용자 선호도/이력 관리)
  - 추천된 식당 클릭 시 네이버 지도 앱/웹 딥링크 연동 처리
  - 배포 후 발견된 프론트엔드 연동 버그(CORS 재발, 상태 꼬임 등) 수정
- **이슈 포인트 (예상):** 네이티브 앱 설치 여부에 따른 딥링크 분기 처리(웹 뷰 브릿지 연동) 이슈 해결

## 4. 일지 작성 공통 가이드라인 (팀원 전체 필독)

1. **증빙 자료 필수 첨부:**
   - 텍스트로만 서술하지 말고, **Postman 테스트 캡쳐, 핵심 소스 코드 스니펫, 에러 로그 화면, 배포 완료된 화면 캡쳐** 등을 반드시 포함합니다.
2. **정량적 수치 사용:**
   - "많이 개선했다" 대신 "API 응답 속도를 500ms에서 120ms로 단축했다", "진행률 85%" 등 수치화하여 작성합니다.
3. **Troubleshooting (이슈 해결 과정):**
   - 개발 중 반드시 1~2개의 에러 상황을 포함하고, **[원인 파악 → 시도한 방법 → 최종 해결책 및 참고문서]** 순으로 논리적으로 서술합니다.
4. **KPT (Keep, Problem, Try) 회고 포함:**
   - 각 차수별 개발일지 하단에 본인의 담당 파트 기준 KPT 회고를 짧게 추가하여 개인의 성장과 다음 단계의 목표를 명확히 합니다.
