* 주요 개발 내용

  1. Redis 캐싱 레이어 적용

    - `ioredis` 클라이언트를 NestJS 모듈로 등록하고 CacheService로 추상화.

    - 식당 조회(GET /api/restaurants): 요청 파라미터(lat, lng, category, radius)를 조합한 캐시 키 생성, TTL 1,800초 적용.

    - 역지오코딩(GET /api/geocode): 위경도 소수점 3자리 반올림 후 캐시 키로 사용, TTL 86,400초 적용.

    - 캐시 히트 시 Naver API 외부 호출 없이 Redis에서 직접 응답, 응답 헤더에 `X-Cache: HIT/MISS` 추가.


  2. Redis 기반 Rate Limit 전환

    - 기존 in-memory 방식의 `express-rate-limit`을 `rate-limit-redis` 패키지로 교체.

    - 서버 재시작 또는 프로세스 재배포 후에도 카운터가 Redis에 유지되어 Rate Limit 우회 불가.


  3. 요청 파라미터 검증 (Validation Pipe)

    - NestJS `ValidationPipe` + `class-validator`를 전역 적용하여 잘못된 파라미터 요청 시 400 Bad Request 반환.

    - radius 허용 범위(500 ~ 2000), category 허용 값('all'|'한식'|'중식'|'일식'|'양식') 검증 규칙 추가.

이슈 관리 및 해결 과정

* 발생한 이슈 (Issue):

  - Redis 캐싱 적용 후 특정 조건(위경도 소수점 6자리 이상)에서 동일 위치임에도 캐시 키가 달라져 캐시 미스가 반복 발생. Naver API 호출 횟수가 예상보다 많아 일일 쿼터 소진 위험이 재등장.


* 원인 분석:

  - 클라이언트(FE)에서 전달하는 위경도 값이 `navigator.geolocation`의 정밀도에 따라 소수점 6~8자리까지 다양하게 변동됨. 동일 위치임에도 소수점 마지막 자리가 달라지면 캐시 키가 다른 값으로 인식되어 매번 새로운 API 호출 발생.


* 해결 과정 및 결과 (Resolution):

  - 캐시 키 생성 시 위경도 값을 소수점 3자리로 반올림(`toFixed(3)`)하도록 `buildRestaurantKey` 메서드 수정. 소수점 3자리는 약 111m 단위로, 서비스 특성상 충분한 정밀도.

  - 역지오코딩 캐시 키도 동일 기준(소수점 3자리) 적용.

  - 결과: 동일 생활권 내 이동 시 캐시 히트율이 크게 향상되었으며, 테스트 환경에서 Naver API 호출 횟수가 기존 대비 약 70% 감소 확인.
KPT 회고 및 차주 계획

* Keep (유지할 점):

  - CacheService를 독립 모듈로 추상화한 덕분에 캐시 키 정책 변경(소수점 자리수 조정 등)이 한 곳의 수정만으로 전체에 반영된 점. 단일 책임 원칙의 효과를 체감.

  - ValidationPipe 전역 적용으로 각 컨트롤러에서 별도 유효성 검사 코드를 작성하지 않아도 되는 구조. 코드 중복이 현저히 줄었음.


* Problem (문제점/아쉬운 점):

  - winston 로거 도입은 완료했으나 로그 파일 로테이션(winston-daily-rotate-file) 설정이 미완성 상태로 로컬 로그 파일이 무한 누적될 수 있는 위험이 존재.

  - EC2 배포 환경이 아직 구성되지 않아 모든 테스트가 로컬 Docker 환경에서만 이루어지고 있음. 실제 네트워크 지연 환경에서의 검증이 필요.


* Try (시도할 점/개선 방향):

  - `winston-daily-rotate-file` 설정을 완성하여 일별 로그 파일 분리 및 보존 기간(7일) 적용.
